# Conductor Handover

## Project Summary

Conductor is a Rust CLI for orchestrating multi-agent workflows between Claude Code and OpenAI Codex. It reads a `workflow.yaml`, executes steps with the correct agent, manages `handover.md` between agents, enforces review gates, and supports snippet triggers like `;;review` and `;;project`.

## Current State

Codebase is clean and functional. All 9 tests pass. `cargo check` clean.

What works:
- `conductor validate` — loads and validates workflow files
- `conductor run` — dry-run mode (default)
- `conductor run --live` — executes real agent-backed steps
- `conductor regen` / `conductor ';;regen'` — analyzes project, asks for custom preferences, writes `workflow.yaml`
- `codex regen` / `codex ';;regen'` — identical behavior, delegates to conductor's regen logic

## Last Session (Claude Code, 2026-04-23)

Code review was performed and three fixes were applied:

1. **Dead loop guard removed** (`runner/mod.rs`) — `HashMap`, `step_counts`, and the inner `> max_steps` check were dead code. The outer `for _ in 0..max_steps` is the sole execution limit.

2. **YAML scalar sanitizer hardened** (`regen.rs`) — `sanitize_yaml_scalar` now escapes `\n`, `\r`, and `\t` in addition to `\` and `"`, preventing malformed YAML for project names with those characters.

3. **`agent.env` wired up** (`runner/agent.rs`) — Both `ClaudeAdapter` and `CodexAdapter` now call `.envs(agent.env.iter())` on their `Command`, so env vars declared in the workflow `agents:` config reach the agent subprocess.

## Open Issues (from review, not yet fixed)

- `;;prefs` snippet is generated by regen but never injected into any step prompt — agents won't see preferences unless they manually type `;;prefs`
- Exit code is swallowed when forwarding to real codex in `src/bin/codex.rs` — should use `exec`/`execv` on Unix
- `AgentConfig.profile` is deserialized but never applied to Codex (Codex reads from `CONDUCTOR_CODEX_PROFILE` env var instead)
- The no-op test `defaults_regen_to_current_directory` still exists
